# BynD Protocol

**The Coordination Layer for veMEZO**

*Aggregate. Direct. Earn. At scale.*

---

## Overview

BynD is a non-custodial coordination layer that aggregates veMEZO voting power, maximizes yield through automated gauge allocation, and issues a liquid ERC-20 token (veBYND) representing a transferable claim on that aggregated position.

Users deposit veMEZO NFTs into BynD once. The protocol maintains those positions at maximum lock and continuously directs voting power to the highest-yielding gauges. In return, users receive veBYND, which can be freely traded or staked to earn protocol yield.

**BynD converts illiquid, inactive governance positions into a liquid, yield-bearing asset.**

---

## The Problem

veMEZO holders control emission direction on Mezo, but participation is structurally low:

- Manual vote management — holders must vote every 7 days or lose all incentives
- Weekly execution overhead — most holders simply do not bother
- Fragmented voting power — individual holders are too small to move gauge outcomes
- No liquidity — veMEZO is a non-transferable NFT with no exit before expiry

As a result: emissions are misallocated, voting power sits idle, and yield is missed every epoch.

---

## The Solution

BynD aggregates veMEZO positions into a single coordinated voting entity and automates all governance actions.

### User Flow

```
01  Deposit veMEZO NFT into ByNdVault
02  Receive veBYND (ERC-20) proportional to deposited voting power — 1:1
03  Optionally stake veBYND to earn rewards
04  Exit position by selling veBYND on the secondary market
```

The underlying veMEZO position remains permanently locked and continuously optimized by the protocol.

---

## veBYND: The Liquid Claim Token

veBYND represents:
- A proportional claim on BynD's aggregated veMEZO voting power
- A share of all rewards generated by that position each epoch

Key properties:
- Fully transferable ERC-20
- No withdrawal or redemption mechanism — the underlying veMEZO stays locked
- Price determined by secondary market liquidity and expected yield

veBYND functions as a non-redeemable, yield-bearing index token backed by permanently locked governance positions.

---

## Reward Token Model

BynD stakers receive rewards from two distinct streams:

### Stream 1 — veMEZO Locking Incentives (MEZO emissions)

Mezo directly rewards veMEZO holders with MEZO token emissions and rebases for keeping their positions locked at maximum duration. These are protocol-level incentives paid by Mezo itself to encourage long-term locking.

When a user deposits their veMEZO NFT into `ByNdVault`, the vault becomes the legal owner of that NFT. This means all MEZO emissions and rebases that Mezo would normally pay to the original holder are now directed to the vault instead.

BynD's design intent is that these MEZO emissions flow through to veBYND stakers proportionally — the vault should claim them and pass them to `ByNdStaking` for distribution.

> **Implementation status:** The MEZO emission claim-and-distribute flow is identified and documented here but not yet implemented in the current contracts. `ByNdStaking` currently only handles one reward token. A multi-token reward extension (MEZO + ERC-20 bribes) is the next planned implementation milestone.

### Stream 2 — Gauge Bribe Incentives (any ERC-20)

Mezo's gauge system pays incentives (bribes) in **any ERC-20 token** — not exclusively MUSD. Protocols that want to attract veMEZO votes to their gauge post incentive tokens directly. BynD harvests these after each epoch vote and distributes them to stakers via `harvestAndDistribute()`.

### Demo and hackathon scope

For this submission, all rewards are processed and displayed as MUSD — Mezo's native stablecoin — using mock contracts that simulate bribe income. This simplifies the demo and keeps the UI readable for judges.

> **Demo note:** All rewards are shown as MUSD. The production protocol supports MEZO emissions (Stream 1) and any ERC-20 incentive token (Stream 2).

### Yield maximization (roadmap)

Because Stream 2 incentives can be any token, identifying the highest-yielding gauge requires pricing each incentive in USD:

```
Yield per vote = (incentive token amount × token USD price) / total veMEZO votes on that gauge
```

BynD's roadmap includes an off-chain keeper script that fetches live token prices, ranks all active gauges by yield per vote, and passes the optimal gauge allocation to `castVotes()` each epoch — turning BynD into a fully automated yield maximization engine.

## Architecture

| Contract | Role |
|---|---|
| `ByNdVault` | Custodies veMEZO NFTs, mints veBYND 1:1, maintains max 4yr lock |
| `VeBYND` | ERC-20 receipt token representing pooled voting power |
| `ByNdStaking` | Distributes rewards proportionally to veBYND stakers |
| `ByNdVoter` | Executes voting, harvesting, and distribution each epoch |

### Epoch Execution

Each epoch consists of three permissionless keeper actions, callable by anyone and incentivized via bounty:

```
Step 1  extendLocks()           →  Reset all veMEZO positions to maximum 4yr lock
Step 2  castVotes()             →  Direct aggregated voting power to highest-yield gauges
                                    (opens 4 hours before epoch end)
Step 3  harvestAndDistribute()  →  Collect all ERC-20 incentives from voted gauges
                                    Pay 1% bounty to caller
                                    Distribute remainder to veBYND stakers
```

## Economic Model

BynD's value is driven by three linked variables:

- **Aggregated Voting Power** — more deposits increase control over gauge emissions
- **Yield Generation** — optimized voting directs incentives toward high-return gauges
- **veBYND Market Price** — reflects expected future yield and secondary liquidity

The system operates as a reflexive loop:

```
Deposits → Voting Power → Yield → veBYND Demand → Deposits  ↺
```

At scale, BynD becomes the dominant veMEZO voter on Mezo — controlling where emissions flow across the ecosystem. This positions BynD as infrastructure, not just a yield product.

This loop is not guaranteed and depends on sustained yield performance and market liquidity.

---

## Liquidity and Exit

BynD does not support withdrawal of underlying veMEZO NFTs. Users exit by selling veBYND on the secondary market. Exit is always available at market price. veBYND may trade at a premium or discount to intrinsic value depending on market depth and demand.

Maintaining deep secondary liquidity is critical to system stability.

---

## Risk Considerations

- Market price deviation due to lack of redemption mechanism
- Dependency on secondary liquidity for efficient exits
- Reliance on external actors to execute epoch functions
- Governance concentration within the protocol
- Incentive token price volatility affecting real yield

BynD is designed for capital efficiency and coordination, not guaranteed price stability.

## Why We Use Two Separate Frontend Folders

During development we discovered a technical limitation with Mezo Passport's wagmi integration that required us to maintain separate frontend configurations for local development versus Matsnet production.

### The Mezo Passport Issue

Mezo Passport provides wallet connectivity through a `getConfig()` function that returns a pre-built wagmi config. This config is designed for Matsnet (Chain ID 31611) and is returned as a **read-only object** — the `chains` array is defined as a non-writable getter. This means it is technically impossible to extend the config with additional chains (such as Hardhat Local, Chain ID 31337) at runtime.

Every attempt to work around this caused one of two failures:
- `TypeError: Cannot set property chains of #<Object> which has only a getter` — thrown immediately when trying to mutate the config
- JavaScript heap out-of-memory — caused by attempting to spread wagmi's full internal chain registry (hundreds of chains) into a new `createConfig()` call

Additionally, Passport's `getConfig()` accesses `window.ethereum` at module evaluation time. In Next.js, modules are evaluated server-side during SSR, where `window` does not exist — causing `ReferenceError: window is not defined` crashes that prevented the page from loading at all.

### Our Solution

We separated the two environments into two provider files:

- `providers-local.tsx` — uses `createConfig()` with `injected()` connector and Hardhat Local chain only. No Passport dependency. Clean, lightweight, no SSR issues.
- `providers-matsnet.tsx` — uses `getConfig()` from Mezo Passport unchanged. Full Passport connector suite, Matsnet chain, wrapped in a dynamic import with `{ ssr: false }` to prevent the `window` crash.

On Matsnet (`bynd_matsnet`), users connect via Mezo Passport normally. On local (`bynd_local`), users connect via MetaMask — and if Passport is installed in the browser, it is still detected as an injected wallet by the `injected()` connector.

## Why We Needed a Local Node for Skip Epoch and Demo Videos

On Mezo Matsnet, epochs last 7 real days. `castVotes()` opens in the last 4 hours of each epoch, and `harvestAndDistribute()` can only be called after voting closes.

This creates a practical problem for development and demo recording: **you cannot demonstrate the full protocol flow in real time on Matsnet without waiting 7 days between each step.**

To solve this, we run a local Hardhat node which gives us full control over the EVM clock. The ⏩ **Skip Epoch** button on the Terminal page (visible only on Chain ID 31337) calls Hardhat's `evm_increaseTime` and `evm_mine` JSON-RPC methods to advance the chain clock:

- **First press** — jumps time to the voting window (4 hours before epoch end), enabling `castVotes()`
- **Second press** — jumps time past the epoch end, enabling the next epoch's `extendLocks()` and `harvestAndDistribute()`

This allows the complete deposit → stake → extend locks → cast votes → harvest → claim flow to be demonstrated in under 5 minutes, while the actual protocol logic — smart contracts, wallet transactions, MetaMask confirmations — is fully real and on-chain.

The Hardhat node comes pre-seeded with 3 mock veMEZO NFTs and 2000 MUSD in the ByNdVoter contract (simulating bribe income), so the harvest produces real yield that flows to stakers immediately.
